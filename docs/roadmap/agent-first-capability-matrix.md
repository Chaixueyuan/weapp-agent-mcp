# agent-first 能力矩阵

目标：面向“agent 主开发、主调试、主测试，人类只做少量兜底”的工作模式，整理当前 `weapp-dev-mcp` fork 的能力现状、下一步优先级，以及每项工作的依赖。

## 状态定义

- `已完成`：当前 fork 已具备，或已在文档/测试项目中验证通过
- `待做`：应进入近期开发队列的能力
- `保留`：方向成立，但暂不进入当前开发批次
- `不做`：当前官方能力受限，或不值得在这一阶段投入

## 分阶段建议

## P0：先保证 agent 稳定工作

### 目标

让 agent 在每次开始操作前，先判断“当前是否健康”，失败时能按统一流程恢复，而不是自己猜测或盲重试。

### 待做项

1. 稳定选择器规范
   - 为业务页面引入统一 `qa-*` 命名
   - 减少 agent 对视觉结构和脆弱 class 的依赖

### 依赖关系

- `mp_recoverConnection` 依赖当前连接管理与日志绑定逻辑
- 稳定选择器规范依赖业务小程序配合，不是 MCP 单侧就能解决
- `mp_healthCheck` 与 `mp_getLogs` 状态增强应共享同一套底层状态模型

## P1：让 agent 稳定判断成功 / 失败

### 目标

让 agent 不再每次临时拼判断逻辑，而是直接调用结构化断言与快照能力。

### 待做项

1. 页面结构快照
   - 汇总 route、query、关键 data、可见元素摘要、局部布局
2. 编译 / 加载状态读取
   - 当前先保留，不进入第一批开发

### 依赖关系

- 断言工具依赖现有 `page_*`、`element_*` 读取能力
- 页面结构快照依赖现有 `currentPage`、`getData`、元素查询、截图 / 局部结构读取能力
- 编译状态读取依赖 DevTools CLI 或 IDE 额外状态通道

## P2：从原子工具走向测试工作流

### 目标

让 agent 能复用一整条场景流程，并输出适合人快速复核的结果，而不是只返回零散日志。

### 待做项

1. `mp_runScenario`
   - 把导航、点击、输入、等待、断言、截图、日志抓取编排为一个流程
2. 回归测试报告产物
   - 用统一 markdown 模板输出结果
3. 专用测试小程序
   - 当前已具备基础样本，后续继续维护覆盖面

### 依赖关系

- `mp_runScenario` 依赖断言工具与页面快照能力
- 回归报告依赖截图、日志、断言结果格式先稳定下来

## P3：让业务项目本身更适合 agent 开发

### 目标

让业务项目本身给 agent 提供更容易进入、切换和复现问题的入口。

### 待做项

1. debug / test 入口
   - 测试账号快速切换
   - mock 数据切换
   - 清缓存 / 重置状态按钮
   - 调试页入口

### 依赖关系

- 强依赖业务项目自身配合
- 不属于 MCP 单独就能闭环的能力

## 推荐的近期开发顺序

1. `mp_healthCheck`
2. `mp_recoverConnection`
3. 稳定选择器规范
4. `mp_getLogs` 状态增强
5. 断言型工具
6. 页面结构快照
7. `mp_runScenario`
8. 回归测试报告

## 当前不建议优先投入的方向

- 继续单纯增加更多点击型原子工具
- 没有稳定选择器前就过早追求复杂场景编排
- 在没有 IDE 状态通道前，把“页面没渲染”一律当成页面逻辑问题
- 直接投入系统原生授权弹窗自动化
